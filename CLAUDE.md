# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

5thPlanet is an **accuracy-first** SEGA Saturn emulator in Rust. The Saturn has eight processors with tightly-coupled timing (2× SH-2 SH7604, MC68EC000, VDP1, VDP2, SCU + SCU-DSP, SCSP M68k + SCSP-DSP, SH-1 CD-block); the project is built up one chip at a time so the foundation stays solid. **Performance is explicitly subordinated to fidelity** — never introduce a JIT, dynarec, or "approximate cycle" shortcut.

The one deliberate exception is the **SH-1 CD-block, which is high-level-emulated (HLE), not low-level-emulated** — its CD-ROM firmware is undumped (on-die mask ROM) and half its job is an analog servo with no observable digital ground truth, so there's nothing to be cycle-accurate *against*. Like every Saturn emulator (MAME/Yabause/Mednafen), we model the host command interface + the buffer/filter/partition engine + the CD-ROM filesystem, reading sectors from a disc image. This is M7 (the HLE-exception decision is **ADR-0015**); see `doc/roadmap.md` and `crates/saturn/src/cd_block.rs`.

**M1–M8 are complete.** M1 (cycle-accurate SH-2 core), M2 (Saturn bus + dual SH-2 + event-driven scheduler), M3 (SCU + SMPC + VDP2 minimal + SCU-DSP + SDL2 scaffolding), M4 (BIOS boots to the SEGA splash — now pixel-matching MAME), M5 (chip-coverage build-out: VDP1 full plotter, MC68EC000 core, full VDP2 NBG/RBG compositor), M6 (SCSP slot/FM audio engine), M7 (the **CD-block**, high-level-emulated — see above — plus the **cartridge slot**), and M8 (**save states + battery-backed backup RAM**). M7's five HLE phases (disc-image loading + TOC/session, the buffer/filter/partition engine, the read pump + data transfer, the ISO9660 filesystem, disc authentication) plus the cartridge slot are done. M8 adds `Saturn::save_state`/`load_state` (bincode snapshot of the whole machine; external media referenced not embedded) and a hardware-faithful, host-persisted internal backup RAM. **M9 (frontend OSD) is ✅ complete (2026-06-11)** — a hand-rolled, software-composited in-window menu (ADR-0008): Esc opens it, with save/load slots, reset, eject/insert disc, **a disc-image browser ("Load Disc…" → navigate the filesystem, pick a `.cue`/`.iso`/`.ccd`, load + boot it)**, quit, and Settings submenus for Graphics (scale 1×–4× + fullscreen), **Controller (press-to-bind keyboard rebind + a live Shuttle Mouse port toggle Off/1/2)**, Region, Cartridge, and **BIOS (power-cycle into a sibling 512-KiB image, save files re-keyed)** — all persisted to a flat TOML-subset **config file** at `$XDG_CONFIG_HOME/5thplanet/jupiter.toml` (`jupiter/src/config.rs`; precedence CLI flag > config > autodetect). **Basic SDL2 GameController support is in** (hot-plug; fixed Xbox-style mapping A/B/C = X/A/B, X/Y/Z = Y/LB/RB, L/R = triggers, left stick = D-pad; controller navigates the OSD); per-button gamepad rebind + analog devices ride with M13 E2. **M10 is done: live physical disc + CDDA→SCSP** — a `SectorSource` trait (image or live drive), CD-audio BGM mixed into the SCSP, and optical-drive reads via the feature-gated `physdisc`/libcdio crate (ADR-0009). Still deferred from M7: the MPEG card and move/copy sector ops.

**M11 (boot a commercial game) is ✅ complete — tag `vf2-good-emulation`**, pursued purely on the **real-BIOS LLE path** — there is no HLE boot. (An opt-in HLE direct boot + HLE BIOS SYS-call library was tried and **removed**; ADR-0010/0011 are Superseded. Rationale: the reference oracle Mednafen is itself LLE, so the only mode in which the master-SH-2 PC-trace-diff methodology works is LLE↔LLE.) Status: the real BIOS authenticates, region-checks, reads IP.BIN, loads the 1st-read program, and jumps to it. **_Doukyuusei ~if~_ is FULLY PLAYABLE** (GFX, SFX, and voices all working — user-verified 2026-06-11) at native 640×224 hi-res — it was the first commercial game to a proper title (「同級生」"PRESS START"), and its in-game record-select **menu renders correctly** (two fixes: SCU indirect-DMA table pointers were read through an unfolded SH-2 cache-through alias → empty descriptors → the menu-bg DMAs moved nothing; and VDP1 framebuffer dots are now horizontally doubled in VDP2 640/704-dot modes; `bfd6240`/`4b93204`). **Virtua Fighter 2 (the original target) is FULLY PLAYABLE at a steady 60 fps** — title (blinking "PRESS START"), mode menu, character select with looping CD-DA BGM, and full 3D fights to the K.O. screen with correct graphics, BGM/SFX balance, and pacing (user-verified 2026-06-10). The 2026-06-10 fix chain that carried it from "stalls in its intro": the stop-then-Play seek origin (`fb52d0c`), the CD periodic clobbering half-composed commands + Get Subcode 0x20 + the BUSY seek-settle (`c1b4dc9`), VDP1 **TVM 8bpp** + DIE-interlace plotting + VDP2 8-bit sprite decode + DD fb-line halving (`0dd3ddd`, savestate v4), the sprite SPCAOS CRAM offset (`5a192c0`), hi-res window-X scaling (`caec91f`), the CD Play **track/index form** (`9d46803`), CD-DA paced at 1× (`b08d100`), and an **SH-2 core bug** — PC-relative fetches are slot-*legal*, with base = branch target + 2 when taken (`1d49088`). Earlier foundations: the CD-block `DCHG`-at-`Init` clear, the SH7604 BCR1 master/slave bit, the FRT `FTCSR` write-0-to-clear bug, VDP2 hi-res rendering, and the `sdbg` slave breakpoint / memory probe / `hirqlog` — all documented in the per-crate sections below and in the `m11-game-boot-progress` memory. The **audio endgame** (post-"playable", same day): SH-2→SCSP **B-bus wait-states** (`7dfbfab` — the region charged 0 vs Mednafen's read +48/write +17, shrinking VF2's sound-submit spin-timeout below the 68k driver's IRQ-masked wake-from-sleep re-init, so the game latched a permanent "sound wedged" flag at `0x060E0017` and silently dropped every SFX request); **CD-DA routed through the SCSP EXTS inputs** (`7ac3837` — see the SCSP bullet; the old full-level sum played BGM 10–20× over the game's programmed mix, drowning the SFX); **KYONEX re-keys only a Release-phase EG** (`e963b19` — a sample-ended slot keeps its EG phase, so a stray strobe must not replay it; the `!active` gate replayed the "Go" announcer on every punch); the fight-loop perf cuts (`03e842a`: SCU interrupt-scan early-out, `dma_pending` guard, cached env flags) and the frontend audio-pacing mirror credit (`021dab5`). Gotcha: a cue/bin whose audio tracks are **byte-swapped (cdrdao MSB-first)** plays CD-DA as noise — the loader now warns (`Disc::audio_looks_msb_first`). See `doc/roadmap.md` for task-by-task status and `doc/glossary.md` for the Saturn-specific vocabulary (chip names, address ranges, register acronyms) used throughout this file.

## Common commands

```bash
cargo check --workspace                    # fastest correctness pass
cargo build --workspace
cargo test  --workspace                    # all unit + integration tests
cargo test  -p sh2                         # just the SH-2 core
cargo test  -p saturn                      # just the Saturn system layer
cargo test  -p sh2 --test opcodes_basic    # one integration test file
cargo test  -p sh2 -- decoder::tests::decodes_branches   # single test by path
cargo clippy --workspace --all-targets -- -D warnings
cargo llvm-cov --workspace --summary-only  # coverage (~85% line; re-report
                                           # from cache: cargo llvm-cov report …)
```

**Formatting:** the codebase is a deliberate compact hand-style — **do NOT run `cargo fmt --all`** (under edition 2024 rustfmt explodes ~60 files; there is no `rustfmt.toml` and no CI fmt gate). Format only the lines you add/change, by hand, matching the surrounding code. `cargo clippy --workspace --all-targets -- -D warnings` *is* the enforceable gate and must stay clean. (See the `code-review` skill for the full rationale.)

Run the binary with `cargo run -p jupiter -- <bios.bin>` — the SDL2 frontend (default-on `sdl2-frontend` feature) opens a window and runs the BIOS; `--no-default-features` runs headless.

### SH-2 core (`crates/sh2/`) — pieces and their contracts

- **`bus::Bus` trait** is the only trust boundary. Each read/write method returns `(value, stall_cycles)`. The host owns wait-state math; the CPU just accumulates. SH-2 is **big-endian** — use `from_be_bytes` / `to_be_bytes` always.
- **`isa::Op`** is one variant per distinct SH-2 encoding (~142 variants). Operand fields (`rn`, `rm`, `imm`, `disp`) are pre-extracted by the decoder so the interpreter never re-parses the raw word. `Op::is_illegal_in_slot()` flags ops that must not appear in a delay slot — extend it when adding new branch/jump/SR-modifying ops. `Op::reads_reg()`, `load_dest()`, `multiply_latency()`, `reads_mac()` drive the pipeline scoreboard; extend them in lockstep.
- **`decoder::decode(u16) -> Op`** is a pure match dispatched on the top nibble, then on the bottom nibble or sub-opcode. Layout mirrors the SH-2 software manual's encoding tables.
- **`interpreter::Cpu::step()`** does interrupt check → fetch → decode → interlock check → execute → cycle-accumulate → scoreboard update. **Delay-slot machinery is centralised here**: when `pending_branch` is `Some`, the next step's instruction is the slot, and PC is overwritten to the branch target *after* the slot retires. Branch opcodes only set `pending_branch`; they never mutate PC directly.
- **On-chip FRT/WDT timers + INTC are event-driven, not per-instruction** (Mednafen's lazy model; M13 A1). `Cpu::step` no longer calls `advance_timers`/`refresh_interrupts` every instruction. Instead: a per-step `if now >= onchip.timer_next_ts()` gate materializes the FRT/WDT only at the next scheduled edge (`OnChip::frt_wdt_update` advances FRC/WTCNT by `(now>>shift)-(lastts>>shift)` — our monotone-`u64` cycle never rebases, so it *is* Mednafen's `ClockDivider`; `frt_wdt_recalc_net` recomputes the edge). A timer-register read/write catches the counters up on demand (`timer_sync_pre`/`_post` around the on-chip blocks in `mem_read*`/`mem_write*`, gated on `OnChip::owns_timer`). The INTC is re-armed **only on change** (`refresh_interrupts` called from `frt_wdt_update` on a flag set, after every on-chip register write, after a DMAC transfer-end, and in `fti_input_capture`) — the signature-gate keeps it idempotent so over-calling is free. **Gotchas:** `OnChip::lastts` is serialized timer phase (savestate v10; `Frt::pre`/`Wdt::pre` removed) and `OnChip::next_ts` **must** serialize too (the lazy FRC field would otherwise diverge on load); `release_slave` calls `reset_timer_epoch` after the cycle resync (a stale `lastts` vs the jumped cycle would over-tick/hang). The gate sits at *end*-of-step to keep the interrupt phase identical to the old model. CKS=3 (external FTCI clock) freezes the FRT.
- **PC-relative addressing uses `instr_pc + 4`**, not the running `regs.pc`. The instruction's own address is plumbed into `execute()` as `instr_pc`; use that for `MOV.L @(d,PC),Rn`, `BRA`, `BSR`, etc.
- **Memory routing** (`Cpu::mem_read*`/`mem_write*`) decodes the SH-2 address into a `(physical, cacheable)` pair via `classify()`. Cached region (`0x00000000..0x1FFFFFFF`) consults the cache (hit = 0 stall; miss = 4 × `bus.read32` line-fill then install). Cache-through (`0x20000000..0x3FFFFFFF`) strips the high 3 bits and bypasses the cache. The **associative-purge space** (regions 2/5, `0x40000000..0x5FFFFFFF` / `0xA0000000..0xBFFFFFFF`) is intercepted before `classify` (`is_assoc_purge`): an access invalidates the matching cache line by address (`Cache::assoc_purge`, masking the region bits so a `0x4xxx_xxxx` purge matches a line installed from `0x0xxx_xxxx`) and does **not** reach the bus (a read returns open bus `!0`) — this is how software drops a stale line for cross-SH-2 coherency (the two caches aren't hardware-coherent). On-chip range (`0xFFFFFE00+`) routes to `Cpu::onchip` instead of the external bus — **except the cache-control register `CCR` at `0xFFFFFE92`, which `mem_read8`/`mem_write8` route to `self.cache` (the cache is a peer field, not part of `OnChip`); without that the BIOS can't enable the I-cache and runs ~8× slow.** **`BCR1` (`0xFFFFFFE0`) bit 15 is the SH7604 MASTER/slave bit** (read-only, a hardware/pin property): the master reads 0, the slave 1. `onchip::Bsc` models it via an `is_slave` flag (set by `Cpu::set_bsc_slave`, which the Saturn calls on the slave after reset — it survives reset). The BIOS cold-start reads it to branch; the slave path skips the work-RAM init, so without it an `SSHON`-released slave re-inits WRAM over a running game (the M11 post-boot crash).
- **`cache::Cache`** stores tag + data per line. `lookup_*` returns `Hit([u8;16]) | Miss | Bypass`; the caller fetches from bus on miss and calls `install`. Writes are write-through (`write_through_u*`) with **no write-allocate** (a write miss never installs a line) — matching the SH7604 and verified against Mednafen (`sh7095.inc` `Cache_WriteUpdate`). `purge()` is the full `CCR.CP` flush; `assoc_purge(addr)` is the by-address single-line invalidate (see memory routing above). **Not modeled:** the cache address-array (`0x60000000`) and data-array (`0xC0000000`) direct-access spaces — they fall through to open bus (rarely used outside cache-as-RAM mode).
- **`exceptions` + `take_exception`** is the single entry point that pushes SR/PC, vectors via `VBR + vec*4`, optionally raises `SR.imask`. Interrupts checked at instruction boundary (never inside a delay slot); illegal and slot-illegal intercepted in `step()` before `execute()`.
- **`harness::MemBus`** is a flat big-endian RAM `Bus` impl for tests. New opcode integration tests under `crates/sh2/tests/` should build CPUs through it rather than introducing parallel bus mocks. `harness::state_digest(cpu, bus, regions)` returns a FNV-1a fingerprint used by the ROM regression harness.

### Saturn system (`crates/saturn/`) — pieces and their contracts

- **`memory::{BiosRom, Ram, StubRegisterBank}`** are typed region structs. Each owns its bytes with big-endian `read*/write*` at *region-local* offsets and folds out-of-range offsets modulo the region size (so any image that's smaller than its window mirrors transparently).
- **`bus::SaturnBus`** impls `sh2::Bus`. Dispatches every access through one `match addr` against `*_BASE..=*_END` region constants in `bus.rs`. Unmapped addresses behave as open bus (0 read, drop write). **Timing is the per-access BSC model (M12 task #8, a faithful Mednafen `BSC_BusRead/Write` port — `BusTiming` + `SaturnBus::charge`):** a **shared bus timestamp** both CPUs sync to (CPU↔CPU arbitration emerges from it, like Mednafen's `SH7095_mem_timestamp`); **CS0 is a 16-bit bus** with per-transaction costs (low WRAM +7, BIOS +8, backup +8, SMPC 0, FTI window +8, unknown +4 — a 32-bit access pays twice); **CS3 high WRAM is 32-bit SDRAM** (read +7, write +2 with a 2-cycle array-busy window the next CS3 access waits out; **cache-line fills are one burst** — `AccessKind::LineFill` beats are free); the **SH-2 write buffer** (a lone store returns 0 stall; only write-after-write backlogs stall); **bus turnaround +1** (back-to-back region change on reads, read→write flips); A-bus cart cost from the live ASR0 wait/strobe fields, CD CS2 +8 per 16-bit; **B-bus exact deferred-write model (M12 #8 residual, `scu.inc BBusRW_DB`):** a write hands off in **+2** CPU cycles and posts its device-side completion (SCSP +17/+13, VDP1 +9/+1 — regs second half free, VDP2 +3/+1 per first/second 16-bit half) on `BusTiming::bbus_write_finish`, which only the *next B-bus access* waits out (CS0/CS3 traffic in between is unaffected); a B-bus **read is always two 16-bit halves** regardless of width (VDP1 +28, VDP2 +40, **SCSP +48** — a 0-wait SCSP let VF2's sound-submit spin-timeout expire before the 68k driver's wake, permanently muting SFX). Per-access stalls are returned **incrementally** (`BusTiming::pay`) so one instruction's several accesses total exactly `mem_ts − cycle`. `AccessKind::Dma` is cost-only on the DMA engines' own timeline, at Mednafen's `dma_time_thing` values: C-bus SDRAM read 6 per 32-bit word / write free, B-bus VDP1/VDP2 **1** and SCSP 13 per 16-bit access; and **a C-bus-endpoint SCU DMA halts BOTH SH-2s for its paced duration** (`RecalcDMAHalt`/`SetExtHalt` — `drain_dma` returns the halting cost and `step_cpus` charges both cores) while a pure A↔B transfer halts neither (instant completion ≈ Mednafen's `CheckForceDMAFinish`; the DMA-end interrupt at trigger-time is the documented boundary). `Saturn::reset` clears `BusTiming` with the CPU cycle counters; serialized since savestate v6 (v9 adds `bbus_write_finish`).
- **`smpc::Smpc`** is the System Manager + Peripheral Control. **Controller ports are selectable** (`PortDevice`: None/Pad/Mouse, default pad-on-1; `Saturn::set_port_devices`): the INTBACK peripheral phase lays out one block per port — pad ID `0x02` (2 active-low data bytes), **Shuttle Mouse ID `0xE3`** (3 bytes: `(flags<<4)|buttons`, X, Y deltas — clamped ±256/255 with overflow flags, Y+ = up, accumulators reset per report; `Saturn::feed_mouse` takes host screen-down Y and negates; M13 E3, savestate v5). The jupiter flag is `--mouse[=1|2]` (port 2 default keeps the keyboard pad; F10 toggles pointer capture, Esc/OSD also releases), also persisted via the frontend config `mouse` key (`off`/`1`/`2`; the flag overrides it) and toggled live from the OSD Controller screen (`set_port_devices` re-points the ports without a reset — the game re-reads devices on the next INTBACK). Register bank at *odd* byte offsets (every other byte reserved). A write to COMREG decodes the byte and queues a `Command` in `pending`; `SF` (status flag) goes busy. The Saturn aggregate drains queued commands between scheduler batches via `take_pending` / `mark_command_done`. `Command` discriminants are `#[repr(u8)]` and **match the hardware codes exactly** — `SshOn = 0x02`, **`IntBack = 0x10`**, `SetTime = 0x16`, **`NmiReq = 0x18`**, `ResEnab = 0x19`, `ResDisa = 0x1A` (these were verified against the SMPC manual; getting INTBACK/NMIREQ swapped silently breaks BIOS boot). **INTBACK is not instantaneous**: it keeps SF busy for a request-dependent time (`intback_busy_us`, reconciled to Mednafen's 4 MHz SMPC-clock model — a status-only INTBACK ≈ 261 µs ≈ 7475 cycles) via `intback_complete_at`, then fills OREG + raises the maskable SMPC interrupt + clears SF — the BIOS polls SF in a wait loop and derails if it clears too early.
- **`vdp2::Vdp2`** is the background generator: register bank + VRAM (512 KiB at `0x05E0_0000`) + CRAM (4 KiB) + a multi-layer compositor (`render_frame` in `vdp2/renderer.rs`). It composites NBG0–3 (tile or bitmap) and RBG0/1 (rotation, via `vdp2/rotation.rs`) by priority, plus the VDP1 sprite layer, with colour calculation, windows, and per-line scroll/zoom/cell-scroll. **Paletted-colour gotchas the BIOS splash exercised:** 8bpp character base is `char × 0x20` (an 8bpp cell is two `0x20` units), the per-layer colour-RAM address offset (`CRAOFA`/`CRAOFB`, `NxCAOS << 8`) selects the CRAM bank, and BGON `NxTPON`/`R0TPON` draw palette code 0 as the *solid* colour `CRAM[offset]` rather than transparent. `Vdp2::owns(addr)` gates bus dispatch. Raster registers (`VCNT`, `TVSTAT` VBLANK/HBLANK/ODD) are **live** — see `Saturn::update_video_timing`. **Hi-res output (M11):** the active resolution is decoded from TVMD (`Vdp2Regs::screen_dims` → 320/352/640/704 wide × 224/240/256, ×2 for double-density interlace); `render_frame` renders that `width × height` (tightly packed, row stride = width) and **returns the dims**. The caller's buffer must be `FRAMEBUFFER_BYTES` = the MAX (`MAX_FRAME_WIDTH 704 × MAX_FRAME_HEIGHT 512 × 4`); the SDL frontend re-creates its streaming texture when the resolution changes and uploads `width × height` at a `width × 4` pitch. `Saturn::run_frame` forwards the dims; the `bios_boot` golden hashes only the active 320×224 region (so it's independent of `MAX_FRAME_*`). **VDP1 always plots at its native horizontal resolution**, so in the 640/704-dot modes each VDP1 framebuffer dot occupies *two* display dots: the sprite-layer sample sites (`sample_sprite`, `sprite_window_pixel`) read the fb through `sprite_framebuffer_x` (`display_x >> 1` when width ≥ 640) — without it a 320-wide sprite layer lands in the left half of a 640-dot frame (was the Doukyuusei menu bug).
- **`vdp1::Vdp1`** is the sprite/polygon engine: a full command-list plotter (`vdp1/plotter.rs`) rasterising textured/scaled/distorted sprites, polygons and lines with gouraud shading and the colour-calc modes, into a double-buffered frame buffer (`0x05C8_0000`) that VDP2 composites. `PTMR` PTM=`0b01` draws once on the write; PTM=`0b10` (automatic) re-renders the list every frame at the buffer swap (`frame_change`).
- **`scsp::Scsp`** is the Sound Processor: 32-slot FM/PCM engine + SCSP-DSP (`scsp/dsp.rs`) + the hosted MC68EC000 (`m68k` crate) in sound RAM at `0x05A0_0000`, released by SMPC `SNDON`. `Saturn::take_audio` drains the mixed 44.1 kHz stereo each frame. **CD-DA enters as the EXTS digital inputs** (M11): `Saturn::run_for` feeds the SCSP exactly the CD samples each batch will consume (`Scsp::cd_need`/`feed_cd`, drawn through the CD-block's pre-roll jitter buffer); `mix` latches one frame per output sample into `dsp.exts`, where (a) **slots 16/17's EFSDL/EFPAN effect-return is the game-controlled CD volume** and (b) the effect DSP reads EXTS as inputs IRA `0x30`/`0x31` and returns the processed mix through other slots' EFSDL (VF2 routes its fight BGM this way with slots 16/17 EFSDL=0). **The per-slot effect-return mixes for EVERY slot, keyed or not** (Mednafen's 32-slot loop: `EFREG[slot]` for 0–15, EXTS for 16/17, 0 for 18–31) — gating it on slot activity silences DSP/CD returns routed through un-keyed slots. **KYONEX key-on is gated on the EG phase, not activity**: a strobe keys a slot iff its EG is in **Release** (`(EnvPhase == RELEASE) == KeyBit`); a non-loop sample running off its LEA only stops fetching (EG keeps its phase), so stale KYONB must not replay it — a key-off moves any non-Release slot to Release, re-arming it.
- **`cd_block::CdBlock`** is HLE (the SH-1 firmware is undumped — see above), modelled on MAME `saturn_cd_hle.cpp`. The host interface (HIRQ/CR1–4, `cmd_pending==0xF` dispatch, periodic report) drives the full engine: the **media** ([`disc::Disc`] — ISO/CUE-BIN/CCD parsers, FAD addressing, TOC); a **200-block pool** + **24 filters/partitions** (FAD-range/subheader matching, true/false routing); a **75 Hz read pump** (`tick` → `play_data` → `read_filtered_sector` → `filter_data`) feeding partitions; **data transfer** (16-bit FIFO + 32-bit SCU-DMA port at `0x0581_8000`); the **ISO9660 filesystem** (`read_new_dir`/`make_dir_current` + the file commands); and **authentication** (`0xE0`/`0xE1`). `insert_disc`/`eject_disc` move the drive between disc-present and empty — **no disc reports `NODISC` (0x07), not `PAUSE`** (matches MAME). **Recognition spin-up (`DrivePhase::Startup`, Mednafen `DRIVEPHASE_STARTUP`, cdb.cpp:2175):** a disc present at power-on/insert reports **`STATUS_BUSY` for ~1 s** (`STARTUP_CYC = cd2m(CD_CLOCK_HZ)`) while the pickup spins up + the TOC is read, *then* settles to PAUSE — `insert_disc` enters `Startup`, `drive_run` settles it, and the `Init` handler is guarded `drive_phase != Startup` so a host Init resets the buffer/filter engine but does **not** park the pickup mid-spin-up. The BIOS fills that window with its **boot animation** (reporting PAUSE immediately skips it). **Disc-change (`DCHG`, HIRQ `0x20`):** `insert_disc` raises the full reset HIRQ set (`0x0BE1`, disc-present only — the no-disc splash keeps MPED-only, so the `bios_boot` golden is safe) and latches `disk_changed`; the host clears `DCHG` write-1-to-clear, **and that W1C also clears `disk_changed`** so the next `Init` doesn't re-raise `DCHG` (matching Mednafen, which clears `DCHG` once during recognition and never re-raises it at `Init`; re-raising it loops BIOS recognition forever). **M10:** sectors are read through a `disc::SectorSource` trait (in-memory `Disc` *or* a live drive via `physdisc`), not the concrete image — `disc` is `Option<Box<dyn SectorSource>>`, `insert_disc` is generic, and reads fill a caller buffer; this is why `SaturnBus`/`CdBlock` are **not `Clone`**. When an **audio** track plays, the read pump decodes 2352-byte sectors to a CD-DA FIFO (`cd_audio`) that feeds the **SCSP's EXTS digital inputs** (`Saturn::run_for` → `Scsp::feed_cd` per batch; the SCSP mixes it at the game-programmed slot-16/17 / DSP levels — see the SCSP bullet). The read pump fills `cd_audio` in **sector bursts** on the scheduler's batch cadence while the SCSP consumes it smoothly per sample, so the feed pulls through `CdBlock::take_cd_audio_buffered`: a **~0.5 s pre-roll jitter buffer** (re-primed only on a fully drained FIFO, with the FIFO cap raised to ~5 s) that absorbs the burst-vs-steady mismatch; the raw `take_cd_audio` (no pre-roll) is kept for the byte-exact decode tests. **Demo/debug hooks (not the BIOS path): `CdBlock::dbg_play_cdda(fad, sectors)` / `dbg_play_first_audio_track()`** drive CD-DA via the real `start_seek` Play machinery without the BIOS issuing Play — the jupiter frontend's **F8** key plays a disc's first audio track live (**debug builds only**, `#[cfg(debug_assertions)]`; release builds omit the F8 binding). **M11: the CD-block drives SCU external interrupt 0** ([`scu::Source::Cd`], IST bit 16, **vector 0x50, level 7**) as a *level* `(HIRQ & HIRQ_Mask) != 0` — `CdBlock::irq_active()`, sampled per master instruction in `step_cpus` and fed to [`Scu::set_cd_int`] (the CD-block can't reach the SCU from inside the bus), masked by IMS **bit 15** (the SCU sign-extends the 16-bit mask), re-armed by the AIACK write (Mednafen `RecalcIRQOut`/`ABusIRQCheck`); VF2 masks it (its GFS polls). MPEG card and move/copy ops remain.
- **`cartridge::Cartridge`** is the rear expansion slot at `0x0200_0000..0x04FF_FFFF` (gated by `Cartridge::owns`). An enum — `None` (empty, floats high to `0xFF`), `Dram` (Extension RAM: two independent mirrored banks at `0x0240_0000`/`0x0260_0000`, ID `0x5A`=1 MB / `0x5C`=4 MB), `Bram` (battery RAM, IDs `0x21`–`0x24`), `Rom` (game ROM at `0x0200_0000`, ID `0xFF`). The cart-ID byte lives at `0x04FF_FFFF`. **All widths compose from `read8_impl`/`write8_impl`** so the backup cart's odd-byte packing (data only in bits 23–16 and 7–0 of each 32-bit word, matching MAME `read_ext_bram`) and the ID-byte placement stay consistent. Plugged in via `Saturn::insert_cartridge` or the frontend `--cart=` flag.
- **`savestate`** (M8; **ADR-0018**) gives `Saturn::save_state() -> Vec<u8>` / `load_state(&[u8])`: a `bincode` snapshot of the whole machine. **Every state type derives `Serialize`/`Deserialize`** — in `saturn` the derive is unconditional; in `sh2`/`m68k`/`scu_dsp` it's behind an optional `serde` feature that `saturn` turns on (keeps the cores dependency-free standalone). Arrays >32 use `serde-big-array` (`#[serde(with = "BigArray")]`), except scu_dsp's `[[u32;64];4]` data RAM (a no_std flat-tuple codec, since big-array is 1-D only). **External media is referenced, not embedded**: `BiosRom.rom`, `CdBlock.disc`, and `Cartridge::Rom.bytes` are `#[serde(skip)]`'d and re-grafted from the live instance in `load_state`; a magic + version header plus FNV-1a BIOS/disc fingerprints reject a load onto the wrong media. Determinism is the contract — the round-trip test boots, snapshots, then runs the snapshot and the original forward by the same budget and asserts identical re-snapshots.
- **`memory::BackupRam`** is the internal 32 KiB battery-backed backup RAM at `0x0018_0000` (the console's built-in memory card). It models the hardware **odd-byte packing** (data only on odd byte addresses, even bytes read 0; `data[(off>>1) % 0x8000]`), matching MAME `backupram_r/w` and the cart `Bram` packing, and is pre-formatted with the "BackUpRam Format" tag (MAME `nvram_init`). `Saturn::internal_backup()`/`load_internal_backup()` expose the raw 32 KiB; the frontend persists it to a `<bios>.bup` file. **Don't revert it to a linear `Ram`** — the packing is load-bearing for backup-manager compatibility.
- **`scu::Scu`** is the System Control Unit. Holds three DMA channels plus timers/IMS/IST/AIACK/ASR/RSEL/VER. DMA trigger fires *only* on 32-bit writes to `D*EN` with bit 8 (DGO) set and non-zero transfer count — byte/halfword writes deliberately don't fire, because software builds the register up piece-by-piece. `take_pending_dma` / `finish_dma` mirror the SMPC drain pattern. The transfer itself runs in `drain_dma`/`scu_transfer` (system.rs). **Every SCU-DMA address is folded to a 27-bit physical address (`& 0x07FF_FFFF`) before the bus access** — games program DMA pointers through SH-2 cache-through aliases (e.g. `0x2600_xxxx` for HWRAM `0x0600_xxxx`), and `SaturnBus` is region-matched, so an unfolded alias reads open-bus 0. This **includes the indirect-mode descriptor-table base pointer** (`req.dst`): folding only the per-triplet src/dst is not enough — an unfolded table base makes every `{size,dst,src}` triplet read as 0 and the walker bail at its safety limit moving nothing (was the Doukyuusei menu-background bug). **SCU→master interrupts are a level, sampled per instruction:** the highest-priority unmasked pending source (`IST & !IMS`) is queried by the master *every instruction* inside `step_cpus` (`take_pending_interrupt`, which clears the acked source's IST and presents the SCU's fixed `0x40 + index` vector — not the SH-2 auto-vector — Phase 2B), so an interrupt lands at the exact instruction `SR.imask` drops below its level rather than at a between-batch drain. **The CD-block external interrupt** (`Source::Cd`, IST bit 16, vector `0x50`, level 7 — M11) is a level set by `set_cd_int` (the aggregate samples `CdBlock::irq_active()` = `(HIRQ & HIRQ_Mask) != 0` each master instruction); the external interrupts are masked by IMS **bit 15** (`take_pending_interrupt` sign-extends the 16-bit IMS, matching Mednafen's `~(int16)IMask`) and re-armed by the AIACK write (0xA8 bit 0; the `cd_prohibit` latch = Mednafen `ABusIProhibit`).
- **`scheduler::SchedEntity`** trait has an associated `Context` so real chips (`SaturnBus`) and test fakes (`()`) can both use it. `next_deadline()` is the global cycle the entity wants to run at; `step(ctx)` advances one unit of work. `Sh2Entity` wraps `sh2::Cpu`; when its `halted` flag is set, `next_deadline()` returns `u64::MAX` so the scheduler's "smallest deadline wins" rule naturally skips it. **Un-halting must resync the cycle**: while halted an entity's `pipeline.cycles` freezes (it's skipped), so releasing it (`Saturn::release_slave` on SMPC `SSHON`) must bump `pipeline.cycles` up to the current `now()` first — otherwise the scheduler sees it as millions of cycles "behind" and runs that many catch-up steps of stale code in one batch ("time travel"). Regression: `dual_sh2::releasing_slave_resyncs_its_cycle_no_time_travel`.
- **`scheduler::Scheduler<E>`** linear-scans `entities` per step. Ties resolve to insertion order — this is the determinism contract for the generic scheduler. Once entity count grows past a handful in M3+, swap the scan for a `BinaryHeap`. **But the live SH-2 pair is no longer stepped by this most-behind rule** (**ADR-0016**) — `Saturn::step_cpus` (system.rs) drives them in Mednafen's **master-leads-slave** order (master one instruction, then the slave catches up to the master's timestamp; Phase 2A), because the most-behind rule could let the *slave* lead and diverged timing-sensitive inter-CPU handoffs. The generic `Scheduler::run_for`/`pick_behind` is retained for the CD-block timer it still steps and for the determinism unit test.
- **`system::Saturn`** owns the bus + scheduler + master/slave IDs. `Saturn::run_for(cycles)` is the headless main loop. Each batch — whose length is **clamped to the next scheduled peripheral-event edge** (`batch_size` → `cycles_to_next_event`: the next VBlank-IN, VBlank-OUT, or pending INTBACK-completion edge, capped by the `SMPC_POLL_QUANTUM = 256` safety ceiling) — steps the two SH-2s in **Mednafen's master-leads-slave order** (`step_cpus`: the master executes one instruction, then the slave runs until it catches up to the master's timestamp; `ss.cpp` `CPU[0].Step` + `RunSlaveUntil` — Phase 2A), with the CD-block timer ticking when the master passes its scheduled cycle. **SCU interrupts are sampled per master instruction inside `step_cpus`** — the SCU IRL is a level the master checks each cycle, so an interrupt lands at the exact instruction `SR.imask` drops below its level (Phase 2B), not once per batch (the former `drain_scu_intc`, now removed). Between batches it calls `update_video_timing()` (derives `VCNT`/`TVSTAT` from the global cycle and raises VBlank-IN/OUT on the raster edges), then `drain_smpc()`, `drain_scu_dma()`, `drain_scu_dsp()`, `drain_vdp1()`, `drain_scsp()`, `drain_input_capture()`. The edge-clamp mirrors Mednafen's `next_event_ts` model (`ss.cpp`), so interrupt assertion + the raster registers settle at the cycle-exact point the reference produces them. **HBlank and SCU-DMA are deliberately NOT clamp edges** (HBLANK is an invented approximation; SCU-DMA is synchronous with no future completion cycle). `run_frame(out)` advances one 263-line NTSC frame in a **single `run_for(CYCLES_PER_FRAME)`** then renders — it must NOT split into active+VBLANK `run_for` calls (the split re-anchors `run_for`'s batch grid and diverges the master's execution from the headless path; that bug stalled VF2). `reset()` halts the slave (matches power-on; BIOS releases it via SMPC `SSHON`) and sets the slave's BCR1 master/slave bit. `drain_input_capture()` applies the inter-CPU FRT input-capture (FTI) pulses the bus flagged this batch (see below). `debug_step_master` / `debug_step_slave` / `debug_drain` are test-only single-step hooks used by the reference-trace diff and `sdbg`.
- **Inter-CPU FRT input-capture (FTI)** — the two SH-2s wake each other via the free-running-timer input-capture pin: a **16-bit** write to `0x0100_0000..0x017F_FFFF` pulses the *slave*'s FTI, `0x0180_0000..0x01FF_FFFF` the *master*'s (`SLAVE_FTI_BASE`/`MASTER_FTI_BASE` in `bus.rs`). The pulse latches FRC→FICR and sets `FTCSR.ICF` via `sh2::Cpu::fti_input_capture` (which first materializes the lazy FRC up to the target CPU's current cycle, then latches — so FICR captures the *current* counter, not a stale event-time value). The bus can't reach the cores, so `write16` just flags `slave/master_input_capture` and `Saturn::drain_input_capture` (plus the per-instruction `apply_fti!` in `step_cpus`) applies it at the aggregate (the SMPC/SCU drain pattern). Games use this to wake a sibling SH-2 parked in an `ICF`-polling loop.

### Borrow patterns to know

- Methods on `Saturn` that need both `&mut self.bus` AND a scheduler entity (e.g. `reset()`) must **destructure `self`** into per-field borrows. Going through the `master_mut()` accessor borrows the whole `Self` and conflicts with the bus borrow. See `Saturn::reset` for the pattern.
- The "queue a side effect, drain at the aggregate" pattern (SMPC + SCU) sidesteps the bus-self-borrow problem: the peripheral notes "I want to fire" in a field; the aggregate later pops the notification (releasing the peripheral borrow) and uses `&mut self.bus` freely.

### Cycle counting

Issue costs returned from `execute()` come from Appendix A of the *SH-1/SH-2 Software Manual*; bus stalls returned from `Bus` are added on top. Don't invent cycle counts — every value should be traceable to a manual entry. Pipeline interlocks (load-use, multiply latency, branch overhead) refine the model further; assertions live in `crates/sh2/tests/pipeline_timing.rs`.

## Project conventions

- **Extend the debugger (`sdbg`) when a debugging task needs a capability it lacks — implement the command/probe first, then debug.** `crates/debugger` (bin `sdbg`) is the primary instrument for boot/CD/render/inter-CPU debugging; treat missing capability as the first thing to fix, not a reason to fall back to ad-hoc trace tests. Add the core hook in `saturn` (a `#[serde(skip)]` debug field + accessor on `Sh2Entity`/`Saturn`) and surface it as an `sdbg` command; keep it observer-only (no core behaviour change) and golden-safe. **For cross-emulator (vs-Mednafen) divergence hunting, the complementary instrument is the config-driven cross-emulator signal "oscilloscope"** (`Scsp::enable_scope`/`take_scope`, the `bios_audio_probe` `SCOPE_PC`/`SCOPE_CH`/`SCOPE_OUT` env, the matching mednaref `sound.cpp` `scope_tick`, and `tools/scope_diff.py`) — the reference emulators (`mednaref`/`mameref`/`yabref`) are **local, never-committed behavioural oracles with no code derived**, the policy recorded in **ADR-0017**: it samples a set of sound-RAM channels on a timebase (a 68k trigger PC → one row per Timer-B tick) with a built-in `t68` 68k-cycle X-axis, on **both** emulators with the same config, and overlays them to report the first divergent row per channel. Prefer it over re-rolling one-off `ENQLOG`/itrace/gate/write-watch probes — it is the generalization of all of them, and the same idea extends to any processor/memory by adding channel sources.
- **Test converage and correctness** -Ensure that corresponding tests are in place whenever code is added, deleted, or modified.
- **Test layout** — opcode tests are integration tests under `crates/sh2/tests/`, one file per instruction family. Saturn-side integration tests live under `crates/saturn/tests/` (one file per peripheral: `bus_routing.rs`, `scheduler.rs`, `dual_sh2.rs`, `smpc.rs`, `scu.rs`, `vdp1.rs`, `vdp2_storage.rs`, `vdp2_render.rs`, `cd_block.rs`, `cartridge.rs`, `savestate.rs`, `bios_boot.rs`). Decoder / pure-module tests live in `#[cfg(test)] mod tests` inside the source file (e.g. the `diagnostics` suite's `all_diagnostics_pass`).
- **Doc comments** — public items in `sh2` should cite the SH-2 manual section they implement when the semantics are non-obvious (delay slots, PC base, SR effects, cycle costs). Public items in `saturn` should call out the SH7604 hardware manual section for the peripheral or memory-map detail being modeled.
- **No `println!`/`eprintln!` in `sh2`** — the crate is `no_std` + `alloc`. Tracing belongs in `debug.rs`.
- **Commits** — Conventional Commits with scopes `sh2` / `saturn` / `jupiter` / `workspace` / `doc` / `ci`. Reference roadmap task numbers when a commit advances the active milestone (e.g. "advances M3 task #2"). Each task that lands also updates `doc/roadmap.md` in a separate `docs:` commit; don't bundle.
- **Temporary files I/O** - if temporary files need to be handled, please use the `tmp/` subdirectory within the project. Avoid using the system's `/tmp` directory, as its limited space may lead to storage issues or write failures.

## Developer tools

The diagnostics beyond `sdbg` (the interactive debugger — see the `crates/debugger` section above). Everything here is **observer-only and golden-safe** — env-gated or a `#[serde(skip)]` debug field — because the rule is *extend the instrument, don't perturb the core*. The strategic tools find *which instruction* diverges from the reference; the watches/harnesses drill into *why*.

- **Self-diagnostics suite (`saturn::diagnostics`).** A built-in battery of checks in **two tiers**, run on **throwaway** `Saturn` instances (never the live machine):
  - **Feature checks** — `run_all() -> Vec<DiagOutcome>`. Each builds a tiny **hand-assembled SH-2 program** (the `diagnostics::asm` encoder — a small superset of the `audio_pipeline.rs` forms; `assemble()` lays reset PC@0/SP@4, code from `0x20`) loaded via `Saturn::new(program)` and run from **reset** — no BIOS, no disc, no toolchain — that writes a result to Work RAM, read back via `sat.bus.low_wram`/`high_wram`. The VDP2/SCSP checks instead drive the chip **through the bus on a CPU-parked machine** (master spins on `BRA .`, slave halted) and inspect a rendered frame / drained audio. Deterministic + golden; the `#[cfg(test)] all_diagnostics_pass` test is the **CI accuracy regression**. ~16 checks across cpu / branch / memory / onchip (DIVU, FRT, on-chip DMAC) / scu (DMA) / vdp2 (back-screen render) / scsp (slot synthesis). **Extend = add a `fn() -> (bool, String)` + one `REGISTRY` entry** (+ a decode assert in `encoder_opcodes_decode_as_expected` if it uses a new opcode form).
  - **System / boot-compatibility checks** — `run_system(bios, disc, region)`. **Heuristic, best-effort (NOT goldens), need real media, NOT in CI/`run_all`**: boots a throwaway and observes `bios_video` (splash non-black), `disc_toc`, and `disc_boots` (the 1st-read program reaches High WRAM = auth + IP.BIN + load + jump worked; early-exits on it).
  - **Surfaces:** `jupiter doctor` (`doctor` = feature checks, exit 0/1; `doctor <BIOS> [disc]` adds the boot checks — region autodetected via `detect_region`, disc via `load_image_disc`); the OSD **Settings → "Diagnostics…"** screen (feature checks via `OsdAction::RunDiagnostics`, run synchronously in `dispatch_osd` on throwaway machines + a live read-only region/disc/master-PC status readout); and the CI test.
  - **Gotcha:** a fresh-reset SCSP has **MVOL = 0 and is silent** (`scsp` reg `0x400`) — the BIOS/driver normally programs it, so any direct-setup audio check (and the `audio_pipeline` test) must write MVOL before expecting output.
- **Cross-emulator trace-diff (the signature LLE↔LLE methodology, ADR-0017).**
  - **Signal "oscilloscope"** — `Scsp::enable_scope`/`take_scope` (`scsp/mod.rs`), driven by `SCOPE_PC`/`SCOPE_CH`/`SCOPE_OUT`/`SCOPE_N` in the `bios_audio_probe` test; samples sound-RAM channels on a 68k-trigger-PC timebase on **both** emulators → CSV. `tools/scope_diff.py` overlays them and reports the first divergent row per channel + sparklines. The generalization of all one-off probes (found the `ADDA`/`ADDX` BGM bug).
  - **`tools/pc_lockstep.py`** (interrupt-aware 68k PC-stream lockstep with cycle deltas) and **`tools/mame_lockstep.py`** (vs MAME's loop-collapsed audiocpu `.tr`).
  - **Master-PC trace generators** (`trace_boot.rs`, `#[ignore]`): `gen_vf2_pc_trace` / `gen_master_pc_trace` / `gen_fullsystem_pc_trace` / `gen_runframe_trace` — loop-collapsed, `PCTRACE_LO/HI`/`PCTRACE_DELAYSLOTS`-filtered dumps to diff vs Mednafen `SS_PCTRACE`. **`sdbg tdiff` hosts this interactively** (stops at the first divergence, rewinds to capture registers).
  - Reference side (gitignored `mednaref`/`mameref`): `SS_PCTRACE[_LO/HI/N]`, `SS_CDTRACE`, `SS_WWATCH`.
- **In-core instruments (`Saturn` debug API, `system.rs`)** — what `sdbg` and the harnesses build on: `enable_master_pc_trace`/`enable_master_pcstream` (PC[+cycle]) / `run_for_traced`; `enable_pctrace[_filtered]`/`take_pctrace` (full reg capture at trigger PCs — a logic analyzer); `enable_seqlog` (cycle-stamped `R[reg]` at a dispatch PC); `enable_raster_jitter` (VCNT/TVSTAT batch-drain jitter, M13 A1); `set_master_bps`/`set_slave_bps` + the cache-vs-truth `bp_probe` + the SCSP-68k `bp68`; `CdBlock::cmd_log` (command ring) + the HIRQ-edge trace; `sh2::Cache::dbg_stats` (hit/miss tally); `debug_step_master`/`debug_step_slave`/`debug_drain`.
- **Bus-level env probes (`saturn`, `OnceLock`-cached, off by default).** `SAT_WWATCH`(+`SAT_WVAL`/`SAT_WWATCH_WIN`) — write-watch at the single chokepoint every writer crosses, tagging the **`AccessKind`** (CPU store vs SCU-DMA vs DMAC vs DSP-DMA) + cycle + PC (the "silently-dropped DMA" instrument); `SAT_RWATCH`(+`_WIN`) — read-watch; `CD_TRACE`/`CD_RWATCH`/`SAT_CDSEEKLOG` — CD command/HIRQ/seek; `SAT_FTILOG` (inter-CPU FRT wake), `SAT_VDP1LOG` (per-command plot), `DMALOG` (SCU DMA).
- **Headless boot/CD harnesses (`trace_boot.rs`, `#[ignore]`, run `cargo test --release … -- --ignored --nocapture`).** The workhorse **`dump_giveup_state`** (`GIVEUP_PC=<pc>`: stop at any boot PC, dump regs + live code + loader-state words + the CD command ring); **`frame_timing`** (slowest `run_frame`s + budget overruns); `dump_cd_state`, `dump_callback_table`, `dump_framebuffer`, `trace_vblank_handler`. (~49 such harnesses live here; most are one-off boot-debug instruments kept around.)
- **Performance benches (`trace_boot.rs`, `#[ignore]`, `--release`).** `bench_fps`, `bench_vf2_fight` (704×448 worst case + audio-samples/frame probe), `bench_vf2_pipeline` (the **only** one that sees thread contention — sequential benches lie), `bench_stages`, `bench_cache`. Basis of the `performance-profile` skill.
- **Frontend diagnostics (`jupiter`).** `SAT_PERFLOG` — per-second EMU/MAIN/audio counters: `frames/s`, the `burst[0/1/2]` distribution (the `burst[2]` attractor signature), `advance avg ms`, present/upload timing, audio-reserve depth. `SAT_FBP`/`SAT_SLAVE_FBP`(+`SAT_FBP_RREG`/`RLO`/`RHI`) — a headless register-window breakpoint for boot debugging in the real binary. Tuning-knobs-as-diagnostics: `SAT_RENDER_THREADS`, `SAT_AUDIO_MS`, `SAT_MAX_BURST`, `SAT_SMP_BATCH`.
- **Disassemblers.** `sh2::debug::disasm` (SH-2) and the standalone MC68000 disassembler in `crates/debugger/src/m68k_disasm.rs`; both surface via `sdbg`'s `d`/`d68`.

## References

The authoritative documents this emulator is built against (cited in per-module doc comments). Treat these as the truth where they're specific; for coupled timing they leave gaps, which is why the behavioral oracle (Mednafen/MAME, ADR-0017) is the reference of last resort.

- **CPUs (fully official, reliably sourced via Renesas/NXP):**
  - **Hitachi SH-1/SH-2 (SH7604) Programming/Software Manual** — SH-2 ISA, exact encodings, and **cycle counts (Appendix A)**. The basis of `crates/sh2/` (cite the section for non-obvious opcode semantics).
  - **Hitachi SH7604 Hardware Manual** — on-chip peripherals (cache, BSC, INTC, DMAC, FRT, WDT, DIVU); the basis of `crates/sh2/src/onchip/`.
  - **Motorola/Freescale MC68000 + MC68EC000 User's Manual** — the SCSP sound CPU (`crates/m68k/`).
- **Saturn chips (SEGA official developer manuals — the primary truth for the system layer):**
  - **VDP1 User's Manual** — sprite/polygon command processor (`vdp1/`).
  - **VDP2 User's Manual** — background/compositor (`vdp2/`; e.g. the `MZCTL` mosaic register).
  - **SCU User's Manual** — System Control Unit + the embedded SCU-DSP (`scu.rs`, `scu_dsp`).
  - **SMPC User's Manual** — System Manager & Peripheral Control; command codes + INTBACK (`smpc.rs`).
  - **ST-V (Sega Titan Video) service/technical manual** — the arcade variant; the **memory-map** reference (`memory.rs` cites its memory-map appendix).
  - **SEGA Saturn Technical Bulletins** — official errata / dev notes.
  - ⚠️ **SCSP** — the official material is thin/incomplete; there is no complete authoritative manual, so the SCSP engine + DSP are modeled on MAME `scspdsp.cpp` + Mednafen rather than a doc (see [ADR-0012](doc/adr/0012-scsp-sound-driver-hle.md) and the `scsp-bios-bgm-audio` memory).
- **SDK conventions (leaked but widely used):** **SGL** (Sega Graphics Library) / **SBL** (Sega Basic Library) programmer's guides — pin down conventions the hardware manuals omit (e.g. the SGL `PER_DGT_*` pad wire-format the SMPC pad layout follows).
- **Behavioral oracles (accurate emulator source, not a document — ADR-0017):** **Mednafen / Beetle Saturn** (`mednaref/`, the game-level reference), **MAME** Saturn/ST-V driver (`mameref/`, low-level mechanics; `saturn_cd_hle.cpp` for the HLE CD-block), **Yabause** (`yabref/`, secondary). Charles MacDonald's Sega hardware notes are the respected community RE reference where they exist.
